Continuous Integration

info

Please do not feel intimidated by the thought of having to make your code pass the CI tests! The core developer team is happy to work closely with contributors to integrate their code and merge PRs.

FlowTorch uses GitHub Actions to run code quality tests on pushes or pull requests to the main branch, a process known as continuous integration (CI). The tests are run for Python versions 3.7, 3.8, and 3.9, and must be successful for a PR to be merged into main. All workflow runs can be viewed here, or else viewed from the link at the bottom of the PR in question.

Workflow Steps‚Äã

The definition of the steps performed in the build workflow is found here and is as follows:

  1. The version of Python (3.7, 3.8, or 3.9) is installed along with the developer dependencies of FlowTorch;
python -m pip install --upgrade pip
python -m pip install flake8 black usort pytest mypy
pip install numpy
pip install --pre torch torchvision torchaudio
pip install -e .[dev]
  1. Each Python source is checked for containing the mandatory copyright header by a custom script:
python scripts/copyright_headers.py --check flowtorch tests scripts examples
  1. The formatting of the Python code in the library and tests is checked to ensure it follows a standard using black;
black --check flowtorch tests
  1. Likewise, the order and formatting of Python import statements in the same folders is checked to ensure it follows a standard using usort;
usort check flowtorch tests
  1. A static code analysis, or rather, linting, is performed by flake8 to find potential bugs;
flake8 . tests --count --show-source --statistics
  1. FlowTorch makes use of type hints, which we consider mandatory for all contributed code, and static types are checked with mypy;
mypy --disallow-untyped-defs flowtorch
  1. Unit tests:

pytest + XML coverage report

pytest --cov=tests --cov-report=xml -W ignore::DeprecationWarning tests/
  1. The coverage report is uploaded to Codecov with a GitHub Action. This allows us to analyze the results and produce the percentage of code covered badge.

If any step fails, the workflow fails and you will not be able to merge the PR into main.

Successful Commits‚Äã

To ensure your PR passes, you should perform these steps before pushing your local commits to the remote branch.

Run Tests‚Äã

Run the tests first so that you can do the code formatting just once as the final step:

pytest tests -W ignore::DeprecationWarning

Fix any failing tests until the above command succeeds.

Check Types‚Äã

Check that there are no errors with the type hints:

mypy --disallow-untyped-defs flowtorch

I find this is one of the most difficult steps to make pass - if you require assistance, comment on your PR, tagging the core developers.

Formatting and Linting‚Äã

Having ensured the tests and docs are correct, run the following commands to standardize your code's formatting:

black flowtorch tests
usort format flowtorch tests

Now, run these commands in check mode to ensure there are no errors:

black --check flowtorch tests
usort check flowtorch tests

It is possible you may need to fix some errors by hand.

Finally, run the linter and fix any resulting errors:

flake8 flowtorch tests

At this point, you are ready to commit your changes and push to the remote branch - you're a star! ‚≠ê From there, your PR will be reviewed by the core developers and after any modifications are made, merged to the main branch.
